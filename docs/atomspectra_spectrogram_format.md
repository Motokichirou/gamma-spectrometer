# Waterfall — референс логики (atomspectra-waterfall) + наш план

> Источник: разбор `shared/spectrum.js` из
> [Maksim-Bartoshyk/atomspectra-waterfall](https://github.com/Maksim-Bartoshyk/atomspectra-waterfall) (MIT).
> **Зачем нам:** это **референс ЛОГИКИ** waterfall — берём *алгоритмы* (биннинг каналов/времени,
> вычитание базового спектра, скользящее среднее, шкалы lin/sqrt/log, рендер heatmap, модель
> «базовый спектр + дельты по интервалам»), чтобы не изобретать.
> **Совместимость форматов — НЕ цель:** у нас своё сервисное ПО и свой внутренний формат записи.
> Формат ниже разобран только чтобы понять их модель данных и алгоритмы.

## Структура файла (построчно, текст, разделитель строк `\n`)

### Базовый спектр (по одному значению в строке), индексы строк с 0:

| Строка | Поле | Тип / смысл |
|---|---|---|
| 0 | `format` | напр. `FORMAT: 3` |
| 1 | `note` | описание |
| 2 | `timestamp` | целое, **эпоха в миллисекундах** |
| 3 | `timestamp2` | вторая метка времени |
| 4 | `lat` | широта (строка) |
| 5 | `lon` | долгота (строка) |
| 6 | `name` | имя спектра |
| 7 | `deviceInfo` | инфо о приборе |
| 8 | `duration` | секунды (float) |
| 9 | `channelsCount` | число каналов (напр. 8192) |
| 10 | `calibrationOrder` = N | порядок полинома калибровки |
| 11 … 11+N | коэффициенты калибровки | N+1 float: `E(ch) = c0 + c1·ch + … + cN·ch^N` |
| 12+N … | counts по каналам | **по одному в строке**, всего `channelsCount` строк |

### Дельты (строки спектрограммы), блоками по **5 строк**

Начало = `11 + (N+1) + channelsCount`. Каждый блок:

| Смещение | Поле | Тип |
|---|---|---|
| +0 | `timestamp` | ms |
| +1 | `lat` | float |
| +2 | `lon` | float |
| +3 | `duration` | сек (float) — длительность этого среза |
| +4 | `channels` | counts всех каналов в **одной строке**, разделитель **TAB** (`\t`) |

## Семантика

- **Дельта = приращение counts за свой интервал `duration`** (не кумулятив). `cps = Σchannels / duration`.
- Биннинг каналов: суммирование соседних; калибровка пересчитывается `c'[i] = binning^i · c[i]`.
- Waterfall: X — каналы (биннинг по умолчанию ×4 → 2048), Y — индекс дельты (время),
  Z/цвет — counts (или cps), палитра Iron, шкала **lin / sqrt / log**, опц. вычитание базового
  спектра и скользящее среднее (верт./гориз.).
- Базовый спектр хранит каналы **по одному в строке**, дельты — **TAB в одну строку** (асимметрия!).

## Какую ЛОГИКУ берём (наш формат — свой)

Модель данных «**базовый спектр + дельты по интервалам**» удобна — берём как внутреннюю
концепцию (в памяти/своём файле), без привязки к их текстовому формату:

- **Срез (дельта)** = `{ timestamp, duration, channels[N] }`, counts — приращение за интервал; `cps = Σ/duration`.
- **Биннинг каналов ×k**: суммирование соседних (калибровка `c'[i] = k^i·c[i]`).
- **Биннинг времени ×m**: суммирование соседних срезов (duration складывается).
- **Вычитание базового спектра** и **скользящее среднее** (верт./гориз.) — опционально, для читаемости.
- **Цветовая шкала**: lin / sqrt / log; **colormap** (`Hot`/`Viridis` ≈ их Iron).

Рендер waterfall — `ImPlot::PlotHeatmap` (грид срезы×каналы). Это и есть всё, что нужно «стащить».

**Температуру/HV** наш сервисный тул пишет в свою структуру среза (`T_pt1000`, `HV_monitor`) —
в их формате этого нет, но нам и не надо под него подстраиваться.

**Один движок захвата на всё:** запись с привязкой ко времени = ровно то, что нужно
термокалибровке (серия срезов timestamp+duration+спектр(+T)); трекинг пика по срезам = кривая
дрейфа `g(T)` = тот же waterfall, только с фитом.

> Калибровочная фаза. На v1 (связь+диагностика) не блокирует — логику зафиксировали, вернёмся при записи/калибровке.
