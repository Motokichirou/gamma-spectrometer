# Протокол shproto / AtomSpectra

<!-- README-I18N:START -->
**Русский** | [English](../en/protocol.md)
<!-- README-I18N:END -->

Прибор говорит по последовательному протоколу **shproto**, совместимому с
[BecqMoni](https://github.com/Am6er/BecqMoni) и [nanopro](https://github.com/Am6er/nanopro).
Перечисляется как виртуальный COM-порт (FT231X) на **600 000 бит/с, 8N1**.

## Формат кадра

```
0xFF  0xFE  [CMD][PAYLOAD …][CRC_LSB][CRC_MSB]  0xA5
 │     │     └──────────── тело со стаффингом ──┘   │
 │     └ START                                      └ FINISH
 └ keep-alive / вводный байт
```

- **`0xFE`** — начало кадра.
- **`0xA5`** — конец кадра.
- **`0xFD`** — байт экранирования.
- **Байт-стаффинг:** любой `0xFE`, `0xFD` или `0xA5` внутри тела передаётся как
  `0xFD` и следом побитовое НЕ (`~byte`) от исходного.
- **CRC16-MODBUS** (`init = 0xFFFF`, `poly = 0xA001`) считается по *нестаффленным*
  `CMD + PAYLOAD`, затем дописывается little-endian и сам стаффится.

## Команды (устройство → хост)

| CMD | Смысл | Полезная нагрузка |
|---|---|---|
| `0x01` | Кусок спектра | `[offset u16 LE][channel u32 LE × N]` (каждое значение маскируется `0x7FFFFFF`) |
| `0x03` | Текст | UTF-8/ASCII текст (ответы на команды, отчёты) |
| `0x04` | Статус | `[elapsed u32][cpu_load u16][cps u32][invalid u32]` |

Заметки:

- Полный спектр на 8192 канала шлётся серией кусков `0x01` (по 512 каналов).
- **Кадр статуса `0x04` триггерит перерисовку спектра** в BecqMoni — слать его после кусков.
- Поле `elapsed` — в **секундах** (BecqMoni трактует через `TimeSpan.FromSeconds`),
  не в миллисекундах.

## Хост → устройство

- Команды шлются текстовыми кадрами `0x03`, например `-sta`, `-sto`, `-rst`, `-inf`, `-cal`, `-tst …`.
- Хост шлёт голый **keep-alive `0xFF`** примерно раз в секунду.

| Команда | Эффект |
|---|---|
| `-sta` | Старт набора · ответ `-ok\r\n` |
| `-sto` | Стоп набора · ответ `-ok\r\n` |
| `-rst` | Сброс спектра · ответ `-ok\r\n` |
| `-stt` | Запрос состояния (AtomSpectra): `-ok collecting\r\n` если идёт набор, иначе `-ok\r\n` |
| `-mode <n>` | Режим прибора (AtomSpectra шлёт `-mode 0` при подключении) · ответ `-ok\r\n` |
| `-inf` | Отчёт о приборе (содержит `VERSION <n>` для AtomSpectra) |
| `-cal` | Энергокалибровка, 40 регистров в формате BecqMoni/AtomSpectra (см. ниже); голый `-cal` — также проверка связи |
| `-cal <i> <hex>` | Записать слово калибровки `i` (0..10), ответ `-ok\r\n` |
| `-tst …` | Встроенный самотест (см. [firmware.md](firmware.md)) |
| `-thr <каналы>` | Порог DSP в каналах (без аргумента — запрос); ответ `-ok thr <каналы>\r\n` |
| `-wcal <order> <c0..cN>` | Записать энергокалибровку человекочитаемо (наш формат), `order = 1..4` |
| `-rcal` | Прочитать калибровку в нашем формате: `CAL: <order> c0 c1 …\r\n<серийник>\r\n` |
| `-calclr` | Стереть сохранённую калибровку |

> **Ack-и команд — строго `-ok\r\n`** (или `-err\r\n`): приложение AtomSpectra сверяет
> ответ ТОЧНО, BecqMoni обрезает по первому `\r` — формат совместим с обоими.

> `-thr`/`-wcal`/`-rcal`/`-calclr` — команды **уровня приложения** нашего прибора и
> сервисного ПО `gammapult`. Формат кадров shproto они не меняют. `-cal` совместима
> с BecqMoni (см. ниже).

## Handshake подключения BecqMoni

Пара особенностей BecqMoni, которые прошивка обязана удовлетворить:

- **Проверка соединения:** BecqMoni шлёт `-cal` и ждёт ровно один текстовый кадр, чьё
  содержимое при split по `\r\n` даёт больше двух строк; предпоследняя строка берётся
  как серийный номер, и устройство загорается зелёным («Connected»).
- **Старт:** при подключении BecqMoni выдаёт `-sto` → `-rst` → `-sta` и ждёт `-ok`
  на каждую.
- Произвольные команды (включая `-tst …`) можно вводить в командной строке на вкладке
  устройства в BecqMoni.

Референсная реализация хоста — в `firmware/tools/becqmoni_sim.py`.

## Энергокалибровка BecqMoni (`-cal`)

BecqMoni умеет читать и записывать энергокалибровку (полином канал→кэВ) прямо в
устройство. Протокол восстановлен реверсом реального обмена и реализован в прошивке
(`cal_store.c`) — кнопки «Записать в устройство» / «Читать с устройства» работают.

**Модель калибровки:** пять коэффициентов `c0..c4` типа IEEE-754 double,
`E(ch) = c0 + c1·ch + c2·ch² + c3·ch³ + c4·ch⁴`.

**Запись** («Записать в устройство») — 11 команд `-cal <i> <hex32>`, `i = 0..10`,
каждая ждёт ответ `-ok`:

| Слова | Содержимое |
|---|---|
| `0,1` | `c0` как double: чётное слово — старшие 32 бита, нечётное — младшие (big-endian) |
| `2,3` | `c1` |
| `4,5` | `c2` |
| `6,7` | `c3` |
| `8,9` | `c4` |
| `10` | контрольная сумма |

Реконструкция double: `u64 = (слово[2k] << 32) | слово[2k+1]`, затем биты как `double`.

**Контрольная сумма (слово 10):** стандартный **CRC-32** (полином `0x04C11DB7`,
reflected `0xEDB88320`, `init = xorout = 0xFFFFFFFF` — обычный zip / `zlib.crc32`),
посчитанный по **ASCII-строке** из слов `0..9`, отформатированных как `%08X`
(верхний регистр, 8 цифр, без разделителей — итого 80 символов).

**Чтение** («Читать с устройства») — та же голая команда `-cal`. Устройство отвечает
**40 регистрами** (как реальный прибор AtomSpectra), по строке через `\r\n`:

| Регистры | Содержимое |
|---|---|
| `0..9` | 5 коэффициентов (пары hi,lo) |
| `10` | CRC-32 |
| `11..38` | прочие регистры (заполнители `00000000`, хостами не используются) |
| `39` | идентификатор устройства (серийник) |

```
40240000\r\n00000000\r\n3FE00000\r\n…\r\n<CRC32>\r\n00000000\r\n…(×28)…\r\nGS474-0001\r\n
```

Удовлетворяет сразу трём потребителям: **BecqMoni** (проверка связи — предпоследняя
строка = серийник; чтение калибровки — слова 0..10), **AtomSpectra** (стартовые
метаданные — ровно 40 токенов, регистр 39 = устройство; чтение калибровки — слова 0..10).

> ⚠ Если в устройстве лежит калибровка с неверной CRC, BecqMoni при пред-чтении бросает
> исключение и закрывает порт. Прошивка хранит слова дословно (запись из BecqMoni) либо
> считает CRC сама (`-wcal`), поэтому сумма всегда верна.
